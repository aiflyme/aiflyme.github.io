02

同源策略 — 为什么恶意网站拿不到 Token

💡

关键不对称：Cookie 由浏览器自动携带（攻击者可利用），但读取另一个域的 Cookie 或 DOM 被同源策略严格禁止。攻击者能"用"你的身份，却看不见那串随机 Token。

03

Token 验证机制 & 安全性分析

✓

核心模型：Token 就是随机钥匙，服务端存原件，客户端提交副本，只做比对。没有推导算法，无从破解。

VerifyCsrfToken 核心源码
PHP 
protected function tokensMatch($request)
{
    // 从 Session 取出服务端存的原始 Token
    $sessionToken = $request->session()->token();

    // 从请求里取客户端提交的 Token（表单字段 或 请求头）
    $token = $request->input('_token') ?? $request->header('X-CSRF-TOKEN');

    // hash_equals：恒定时间比较，防止时序攻击
    return is_string($sessionToken)
        && is_string($token)
        && hash_equals($sessionToken, $token);
}
为什么极难破解 — 四层保障
密码学随机性

random_bytes(40) = 320 位随机数，2³²⁰ 种可能。每秒猜 10 亿次，所需时间远超宇宙年龄

绑定 Session，每人不同

每个用户 Session 独立生成，Session 过期即失效，猜中一人的 Token 对其他用户无效

hash_equals() 防时序攻击

普通 == 遇到第一个不同字符就返回，攻击者可通过响应时间逐字试探。hash_equals 强制走完整比较，耗时恒定，时序信息为零

Token 每次请求后轮换

成功响应后重新生成，旧 Token 立即失效。即使截获旧 Token 也无法复用

04

Cookie 与 Session 本质区别 & 认证协作流程

💡

Cookie 是信使，Session 是保险箱。Cookie 只负责把 Session ID 带给服务器，真正的数据锁在服务器的 Session 里。

维度CookieSession
数据存在哪浏览器本地服务器（文件 / Redis / DB）
容量4 KB无限制（服务器决定）
客户端可见是（加密后）否，只暴露随机 ID
自动携带每次请求浏览器自动带ID 通过 Cookie 带，数据留服务端
关浏览器后无 Expires 则消失ID Cookie 消失，服务端数据按 lifetime 保留
Cookie 重要安全属性
属性作用Laravel 默认
HttpOnly禁止 JS 读取，防 XSS 盗取 Cookie已开启
Secure只在 HTTPS 连接下发送需手动开启
SameSite=Lax阻止第三方跨站请求携带，辅助防 CSRFLax
Expires/Max-Age持久化 Cookie；不设则关闭浏览器失效由 expire_on_close 控制
Domain/Path限制 Cookie 作用范围，避免子域泄露仅当前域
三种"关闭"的行为对比
操作Cookie（无 Expires）SessionlocalStorage
关闭标签页还在还在还在
关闭浏览器消失ID 消失还在
清除浏览器数据消失消失消失
换浏览器消失消失消失
换设备消失消失消失
💡

关闭浏览器只是关掉进程，localStorage 数据写在本地磁盘，重新打开还在。Session 的 ID Cookie 消失了，但服务端数据还在，只是找不到入口了。

Session 不一定依赖 Cookie — 三种传递方式
Cookie 携带 Session ID（最常用）

Web 应用标准方案，HttpOnly + Secure 保护，Laravel 默认方式

URL 参数携带（不推荐）

?session_id=abc123，会出现在浏览器历史、服务器日志、Referer 头中，安全性极差

请求头 JWT Token（API / SPA）

Authorization: Bearer token，服务端完全无状态，不用 Session 也不依赖 Cookie，Laravel Sanctum Token 模式

// Blade 里读闪存

Session 配置（config/session.php + .env）
PHP + .env 
// config/session.php
'driver'          => env('SESSION_DRIVER', 'file'),
// 可选: file | redis | database | cookie | memcached

'lifetime'        => 120,    // 服务端 Session 数据保留分钟数
'expire_on_close' => false,  // true = 关浏览器立即过期
'http_only'       => true,   // JS 读不到（默认已开）
'secure'          => true,   // 只在 HTTPS 发送（生产必须开）
'same_site'       => 'lax',  // 跨站请求限制

// .env 生产推荐配置
SESSION_DRIVER=redis
REDIS_HOST=127.0.0.1
SESSION_SECURE_COOKIE=true
⚠

存储在哪（driver）和何时过期（lifetime / expire_on_close）是两个完全独立的配置，任意组合。

05

Session ID 会被窃取吗

攻击方式手段防御
XSS 注入脚本document.cookie 读取后外传，最常见HttpOnly Cookie
网络嗅探 / 中间人HTTP 明文截获，公共 WiFi 高危Secure Cookie + HTTPS
共享设备 / 物理访问直接查看或复制 Cookie 文件短 lifetime + 登出销毁
子域名 Cookie 泄露evil.yourapp.com 读取父域 Cookie精确 Domain + SameSite=Strict
会话固定攻击预先植入 Session ID，等用户登录后复用登录后 regenerate()
✓

HttpOnly + Secure + HTTPS 三件套加合理过期策略，已覆盖绝大多数真实攻击场景。剩余风险本质是 XSS 和物理安全问题，不是 Session 机制本身的缺陷。

访客购物车推荐可但容量小最常用纯前端方案
Session 方案（主流）
PHP 
// 访客加购
Session::push('cart', [
    'product_id' => 101,
    'qty'        => 2,
    'price'      => 99.00,
]);

// 登录后合并到数据库
foreach (Session::get('cart', []) as $item) {
    Cart::updateOrCreate([
        'user_id'    => auth()->id(),
        'product_id' => $item['product_id'],
    ], ['qty' => DB::raw('qty + '.$item['qty'])]);
}
Session::forget('cart');
localStorage 方案（前端驱动）
JavaScript 
// 加购
const cart = JSON.parse(
    localStorage.getItem('cart') || '[]'
);
cart.push({ product_id: 101, qty: 2 });
localStorage.setItem('cart', JSON.stringify(cart));

// 结账时同步服务端
fetch('/api/checkout', {
    method: 'POST',
    body: JSON.stringify({ items: cart })
});
localStorage 方案优缺点
关闭浏览器数据不丢失

数据写在本地磁盘，重新打开浏览器购物车依然存在，体验优于 Session 方案

不占服务器资源，容量大

5MB+ 存储空间，无需服务端 Session 表或 Redis，轻量级项目首选

换设备购物车消失

换浏览器、换手机、重装系统后数据全无，需要登录后同步到服务端数据库

价格数据不可信

JS 可直接修改 localStorage 里的价格字段，服务端结账时必须忽略前端传来的价格，重新查库验证

⚠

无论哪种方案，结账时服务端必须重新查数据库验证价格和库存，绝不能相信前端传来的价格数据。

07

Session 持久化 & 数据库多行问题

为什么用 Redis / 数据库存 Session
多台服务器的问题

文件/内存存储：用户第1次打到服务器A建了Session，第2次请求负载均衡到服务器B → B没有这个Session → 被踢出登录

Redis 共享存储解决问题

所有服务器读写同一个 Redis，Session 数据共享，用户无感知

Redis TTL 自动清理

按 lifetime 自动过期删除，无需额外任务。数据库方案需定期执行 php artisan session:gc

数据库出现 2 行的原因
原因说明判断方法
最常见 登录触发 regenerate() 访客 Session 行残留 + 登录后新行创建，旧行等 GC 旧行 user_id = NULL
多浏览器同时登录 Chrome 一行，Firefox 一行，两行都有效 两行 user_id 都有值
过期数据未 GC GC 概率触发（默认 2/100），旧行未及时清理 旧行 last_activity 很早之前
SQL 
-- 快速排查
SELECT id, user_id, FROM_UNIXTIME(last_activity) AS time
FROM sessions
ORDER BY last_activity DESC;
💡

user_id = NULL 的行是访客遗留，等 GC 自动清理。手动立刻清理：php artisan session:gc
